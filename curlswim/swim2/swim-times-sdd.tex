\section{Introduction}
% =================================================================

\subsection{Purpose}

This Software Design Document (SDD) describes the architecture
and detailed design of the \textbf{swim-times} program.  It
satisfies the design-description function of IEEE~Std~1016-2009
by presenting a set of design views, each tailored to a specific
stakeholder concern.  Together with the Concept of Operations
(CAST-CONOPS-001) and the Software Requirements Specification
(CAST-SRS-001), this document forms the complete development
baseline for swim-times.

\subsection{Scope}

This SDD covers all code that is tangled out of the literate
program \code{swim2/swim-times.w}.  It does not cover the
build system, deployment, or the companion Pandoc-based PDF
report (\code{swim/swim-times.md}); those are described
elsewhere.

\subsection{Relationship to the Parnas--Clements model}

Per Parnas and Clements (1986), this document presents the
design as if it had been derived top-down from requirements,
even though in practice the program evolved through exploratory
prototyping against an undocumented third-party API.  Each
design decision is presented in the form a reader would find
most useful, with the rationale recorded so that future changes
can be evaluated against the original intent.

\section{Design Approach}
% =================================================================

\subsection{Architectural style}

swim-times is a single-process, single-threaded
\emph{pipes-and-filters} architecture realised within one address
space: command-line options flow into an option-parsing stage,
which feeds a swimmer-resolution stage, which feeds a per-event
fetch-parse-sort stage, whose output is serialised by a
formatter to standard output.  Each stage is a CWEB chunk
(or small group of chunks) with a single responsibility.

\subsection{Key design decisions}

\begin{description}[style=nextline]
  \item[D-1: CWEB literate source.]
        The canonical source is \code{swim-times.w};
        \code{swim-times.c} is a build artefact.  This satisfies
        \req{REQ-D-02} and \req{REQ-A-04}.

  \item[D-2: 24-line chunk discipline.]
        No CWEB chunk exceeds twenty-four lines.  Larger logical
        units are decomposed into named sub-chunks.  This satisfies
        \req{REQ-D-03} and improves readability for the typeset
        documentation.

  \item[D-3: String-scanning JSON parser.]
        A purpose-built scanner (\code{scan\_string},
        \code{scan\_long}) is used in place of a third-party
        JSON library.  This satisfies \req{REQ-D-04} and is
        adequate because the Sisense JAQL response shape is
        stable and well-understood.

  \item[D-4: Static bearer token.]
        The Sisense bearer token is a compile-time constant.
        Token rotation is a maintenance event, not a runtime
        operation; this matches \req{REQ-A-02}.

  \item[D-5: Insertion sort.]
        Per-event row counts rarely exceed a few dozen, so
        $O(n^2)$ insertion sort is used in preference to a more
        complex algorithm.  Simpler code, fewer failure modes.
\end{description}

\section{Design Views}
% =================================================================

% --------------------------- 4.1 ---------------------------------
\subsection{Context view}

\paragraph{External actors.}
\begin{itemize}
  \item \textbf{Operator} --- a coach, parent, or maintainer
        invoking the binary from a shell.
  \item \textbf{Sisense JAQL endpoint} --- the upstream HTTPS
        service at \texttt{usaswimming.sisense.com}.
  \item \textbf{Standard output / standard error} --- the
        downstream sinks that receive table or CSV output and
        diagnostics, respectively.
\end{itemize}

\paragraph{Trust boundary.}
The only trusted external system is the Sisense endpoint, which
serves only public competitive records.  No user-supplied data
crosses a trust boundary; command-line arguments are consumed
internally.

% --------------------------- 4.2 ---------------------------------
\subsection{Composition view}

The program is decomposed into the following top-level CWEB
chunks (the order in which they appear in the tangled output):

\begin{center}
\begin{longtable}{ll}
  \toprule
  \textbf{Chunk} & \textbf{Responsibility} \\
  \midrule\endhead
  \code{Includes}                 & Standard headers and \code{<curl/curl.h>}       \\
  \code{Global constants}         & Numeric limits, Sisense token, event table     \\
  \code{Option flags}             & \code{OPT\_*} bitmasks and global option state \\
  \code{Type definitions}         & \code{Buffer}, \code{TimeRow}, \code{Swimmer}
                                    structs                                          \\
  \code{HTTP utilities}           & \code{write\_cb}, \code{post\_json}            \\
  \code{JSON scanner}             & \code{scan\_string}, \code{scan\_long}         \\
  \code{Database pipe}            & \code{db\_psql\_command}, \code{db\_open},
                                    \code{db\_close}, \code{csv\_emit\_field}        \\
  \code{Person lookup}            & \code{lookup\_person\_key}                     \\
  \code{Times fetch}              & \code{insertion\_sort}, \code{format\_date},
                                    \code{fetch\_times}                            \\
  \code{Offline fetch}            & \code{offline\_split}, \code{offline\_fetch}    \\
  \code{Main function}            & Option parsing, swimmer table, \code{main}     \\
  \bottomrule
\end{longtable}
\end{center}

Each top-level chunk is itself a composition of named sub-chunks
to honour the 24-line discipline (D-2).  For example,
\code{post\_json} expands to \code{Initialize curl handle} +
\code{Issue HTTP request and return}; \code{fetch\_times} expands
to five sub-chunks corresponding to URL build, body build, fetch,
parse, and print.

\begin{center}
\begin{tabular}{ll}
  \toprule
  \textbf{Caller}       & \textbf{Callee}                              \\

  \code{main}           & \code{db\_open}, \code{db\_close} (under \code{store}) \\
  \code{main}           & \code{lookup\_person\_key}, \code{fetch\_times} (online) \\
  \code{main}           & \code{offline\_fetch} (under \code{offline})            \\
  \code{lookup\_person\_key} & \code{post\_json}, \code{scan\_string}, \code{scan\_long} \\
  \code{fetch\_times}   & \code{post\_json}, \code{scan\_string},
                          \code{insertion\_sort}, \code{format\_date},
                          \code{csv\_emit\_field}                        \\
  \code{offline\_fetch} & \code{popen}, \code{offline\_split},
                          \code{insertion\_sort}                          \\
  \code{db\_open}/\code{db\_close} & \code{popen}/\code{pclose} +
                          \code{psql} child process                       \\
  \code{post\_json}     & \code{libcurl} easy interface                  \\
  \code{post\_json}     & \code{write\_cb} (callback)                    \\
  \bottomrule
\end{tabular}
\end{center}

The dependency graph is acyclic.  The only linked external
dependency is \code{libcurl}; Postgres is reached through a
child \texttt{psql} process spawned by \texttt{popen(3)}, so no
\texttt{libpq} is statically required.  Furthermore the host is
not assumed to provide a \texttt{psql} client at all: by default
\code{db\_psql\_command} expands to \texttt{../../pg/pg.sh psql},
which executes \texttt{psql} \emph{inside} the Postgres container
via \texttt{docker exec}.

% --------------------------- 4.4 ---------------------------------
\subsection{Information view (data structures)}

\paragraph{\code{Buffer}.}
A growable heap buffer used to accumulate libcurl response chunks.
Always null-terminated so it can be treated as a C string.

\begin{lstlisting}
typedef struct {
    char  *data;
    size_t size;
} Buffer;
\end{lstlisting}

\paragraph{\code{TimeRow}.}
One swim-time record.  \code{sort\_key} is the Sisense numeric
sort key parsed as a \code{double}; the remaining fields are
fixed-size character arrays sized to comfortably exceed the
longest values observed in production data.

\begin{lstlisting}
typedef struct {
    double sort_key;
    char   time[32];
    char   date[16];
    char   standard[48];
    char   meet[256];
} TimeRow;
\end{lstlisting}

\paragraph{\code{Swimmer}.}
A static configuration record describing how to identify each
tracked swimmer in the Sisense Persons table and, optionally,
the inclusive ISO date window to which results are restricted
(used by \code{ledecky10}).

\begin{lstlisting}
typedef struct {
    const char *search_query;  /* Sisense person-search "contains" */
    const char *match_substr;  /* lowercase substring filter       */
    int         flag;          /* OPT_STELLA, OPT_KALEA, ...       */
    const char *date_min;      /* "YYYY-MM-DD" inclusive (or NULL) */
    const char *date_max;      /* "YYYY-MM-DD" inclusive (or NULL) */
} Swimmer;
\end{lstlisting}

\paragraph{Database schema (\texttt{schema.sql}).}
Persisted rows live in the \texttt{swim\_times} table, whose
columns mirror the seven fields written by the DB-pipe emitter
plus an identity primary key and a \texttt{fetched\_at}
timestamp.  A UNIQUE constraint on
\texttt{(swimmer, event, swim\_time, swim\_date, meet)} prevents
duplicate inserts on repeated \code{store} runs (REQ-F-73).
Two indexes (\texttt{(swimmer, event)} and
\texttt{(event, sort\_key)}) accelerate the \code{offline}
\texttt{SELECT} (REQ-F-71) which filters on swimmer name and
event code and orders by \texttt{sort\_key}.

\paragraph{Global state.}
Three module-level variables hold parsed command-line state:

\begin{itemize}
  \item \code{g\_opts} --- bitmask of \code{OPT\_*} flags.
  \item \code{g\_events[NUM\_EVENTS][32]} --- user-requested
        event codes.
  \item \code{g\_nevents} --- count of valid entries in
        \code{g\_events}.
\end{itemize}

% --------------------------- 4.5 ---------------------------------
\subsection{Interface view}

\paragraph{Command-line interface.}
\begin{itemize}
  \item \code{-o keyword[,keyword\ldots]} --- swimmer and output options.
  \item \code{-e event[,event\ldots]} --- restrict to event codes;
        repeatable.
\end{itemize}

\paragraph{HTTP interface.}
Two POST endpoints are consumed (skipped under \code{offline}):
\begin{itemize}
  \item \texttt{.../aPublicIAAaPersonIAAaSearch/jaql}
        --- person search.
  \item \texttt{.../aUSAIAAaSwimmingIAAaTimesIAAaElasticube/jaql}
        --- times query.
\end{itemize}

Both carry the headers \code{Content-Type: application/json} and
\code{Authorization: Bearer \{SISENSE\_TOKEN\}}.

\paragraph{Database interface.}
Postgres is reached only through a child \texttt{psql} process
launched by \texttt{popen(3)} (REQ-I-12, REQ-I-13).  The shell
prefix is taken from the environment variable
\texttt{SWIM\_TIMES\_PSQL} when set, otherwise the default
\texttt{../../pg/pg.sh psql} is used --- the \texttt{pg.sh}
wrapper, which runs \texttt{psql} inside the container via
\texttt{docker exec}, so the host need not have a \texttt{psql}
client of its own.  Two access patterns are needed:
\begin{itemize}
  \item Write (\code{store}): a single
        \texttt{COPY swim\_times (\dots) FROM~STDIN WITH (FORMAT csv)}
        invocation streamed for the lifetime of the run.
  \item Read (\code{offline}): one
        \texttt{psql -t -A -F'|' -c 'SELECT \dots'} per event,
        emitting pipe-separated rows that the parser splits on
        the \texttt{|} character.
\end{itemize}

\paragraph{Standard output.}
Either a CSV stream (header line followed by data lines) or a
plain-text table (per-event heading, column header, separator,
data rows, blank line).

% --------------------------- 4.6 ---------------------------------
\subsection{Interaction view}

A typical invocation flows through the following sequence:

\begin{enumerate}
  \item \code{main} parses \code{-o} / \code{-e} via \code{getopt}.
  \item Unless \code{offline} is set, \code{main} initialises
        libcurl (\code{curl\_global\_init}).
  \item If \code{store} is set, \code{main} calls \code{db\_open}
        which spawns a \texttt{psql} child running a
        \texttt{COPY \dots\ FROM~STDIN} transaction.
  \item For each entry in \code{SWIMMERS} not masked out by
        \code{g\_opts}, the dispatcher chooses the online or
        offline path:
        \begin{enumerate}
          \item \emph{Online.} \code{lookup\_person\_key} issues
                one POST and returns the \code{PersonKey}.  For
                each requested event (or all default events),
                \code{fetch\_times} issues one POST, parses the
                response into a \code{TimeRow} array, applies the
                \code{Swimmer}'s optional date window, sorts the
                rows, and emits them to stdout, the DB pipe, or
                both per the option mask.
          \item \emph{Offline.} \code{offline\_fetch} runs
                \texttt{psql -c 'SELECT \dots'} for each event,
                parses pipe-separated rows back into a
                \code{TimeRow} array, applies the date window,
                sorts, and emits.  No HTTP is performed.
        \end{enumerate}
  \item \code{main} calls \code{db\_close} (no-op when no DB
        pipe was opened) and, unless \code{offline}, also calls
        \code{curl\_global\_cleanup} before exiting.
\end{enumerate}

% --------------------------- 4.7 ---------------------------------
\subsection{Algorithm view}

\paragraph{JSON scanning.}
\code{scan\_string} locates the next occurrence of the literal
\code{"key"}, advances past the colon and any whitespace, expects
a double-quote, and copies bytes until the matching closing quote
into the output buffer.  \code{scan\_long} is the integer
counterpart, using \code{strtol} on the digits following the colon.
Both functions advance an in/out position pointer so that
sequential scans walk forward through the response without rescan.

\paragraph{Sort.}
\code{insertion\_sort} performs an in-place ascending sort of a
\code{TimeRow} array by \code{sort\_key}.  Chosen over heap or
quick sort because $n$ rarely exceeds 50 and insertion sort has
no recursion, no auxiliary memory, and trivial correctness.

\paragraph{Date reformat.}
\code{format\_date} converts \code{YYYYMMDD} (eight bytes) to
\code{YYYY-MM-DD} (ten bytes plus terminator) by direct byte
copy and hyphen insertion---no \code{sscanf} round-trip required.

% --------------------------- 4.8 ---------------------------------
\subsection{Resource view}

\paragraph{Memory.}
Per-event peak memory is bounded by
$\code{MAX\_TIMES} \times \mathtt{sizeof}(\code{TimeRow})$
(approximately 70~KiB per event), plus the response buffer
(typically 30--80~KiB).  Buffers are freed at end of each
\code{fetch\_times} call; no long-lived heap state accumulates
across events.

\paragraph{Network.}
Each invocation issues at most $4 + (4 \times 31) = 128$ HTTPS
POSTs.  All connections are short-lived; libcurl manages connection
reuse internally where the easy-interface allows.

\paragraph{Credentials.}
The Sisense bearer token is a static const string in the binary.
Token rotation requires editing \code{swim-times.w} and
recompiling.  No filesystem-level credential storage is used.

\section{Design Rationale}
% =================================================================

\subsection{Why CWEB?}

CWEB satisfies \req{REQ-A-04} (self-documentation) directly: the
typeset \code{swim-times.pdf} is the program's reference
documentation.  Equivalent C-with-comments would have required
maintaining design prose in a separate document, with the
inevitable drift between code and documentation that the
Parnas--Clements paper warns against.

\subsection{Why no JSON library?}

Adding \code{cJSON} or \code{jansson} would introduce a build
dependency and would obscure the small, focused parsing logic
that actually serves the program.  The Sisense response shape is
stable and the scanner has fewer than fifty lines of code; the
cost of a custom scanner is therefore lower than the cost of a
third-party dependency.

\subsection{Why static bearer token?}

The token is read-only against a public dataset; embedding it
matches the actual security posture (low) and avoids the
operational complexity of credential injection.  When USA~Swimming
rotates the token (an event observed only twice in the past two
years), the maintainer recompiles and redeploys.

\subsection{Why insertion sort?}

\req{D-5} addresses this directly.  The complexity argument is
not just about wall-clock time---it is about reading the code:
twelve lines of insertion sort can be verified by eye, whereas a
heap sort cannot.  For $n < 100$ the wall-clock difference is
imperceptible.

\subsection{Why one binary, not two?}

A separate person-search utility was considered but rejected.
The two queries (person search, times query) are always issued
together, and splitting them would force the caller to thread
the \code{PersonKey} through the shell, complicating every
invocation for no gain.

\subsection{Why \texttt{psql} via \texttt{pg.sh}, not \texttt{libpq}?}

The host build environment is not assumed to carry either
\texttt{libpq} headers \emph{or} a \texttt{psql} client --- the
project's macOS development host has neither installed.  What it
does have is the \texttt{pg.sh} wrapper alongside a Postgres
Docker container; \texttt{pg.sh psql} runs \texttt{psql}
\emph{inside} that container via \texttt{docker exec}, so the
client binary ships with the database itself.  Spawning
\texttt{pg.sh psql} through \texttt{popen(3)} therefore
(a)~keeps the swim-times binary's link-time dependency surface
unchanged (\req{REQ-D-04}: still only \texttt{libcurl} + libc),
(b)~removes the host \texttt{psql} assumption entirely
(\req{REQ-I-13}), and (c)~still uses Postgres's optimised
\texttt{COPY \dots\ FROM~STDIN} bulk path on the write side.  The
cost is one \texttt{docker~exec} per run for \code{store} and one
per event for \code{offline} --- both negligible against the
100~ms-class network latencies dominating online runs.

\subsection{Why a date-window field on \texttt{Swimmer}?}

\code{ledecky10} demands age-bracket filtering that the
USA~Swimming JAQL endpoint does not directly express.  Storing
the date window on the \code{Swimmer} record (rather than as a
separate command-line flag) keeps the per-swimmer policy in one
place and makes new historical comparisons additive: another
swimmer with a different age window can be added by appending a
single row to \code{SWIMMERS} with no change to \code{main}.
